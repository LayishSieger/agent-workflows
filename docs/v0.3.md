# agent-workflows v0.3 — design freeze

Status: **0.3** — policy-driven shared tick, dual schedulers, host-workflows.  
Source: [Spec #10](https://github.com/LayishSieger/agent-workflows/issues/10) and wayfinder decisions [#1](https://github.com/LayishSieger/agent-workflows/issues/1)–[#7](https://github.com/LayishSieger/agent-workflows/issues/7).

This freeze is the perimeter for 0.3. Loop rewrite, host package, and init offer are implemented against this shape; **this document** remains the shared design source of truth.

## Scope

| In 0.3 | Out of 0.3 |
|--------|------------|
| Full tracker **ops contract** in product policy (named ops + Input/Steps/Success/Failure) | Local/Linear full runtime proof (same headings + stubs OK) |
| Shared tick in `loop-workflows` (tracker-agnostic ops names only) | Park-branch fleet; multi-spawn; rate limits |
| Dual schedulers: chat (`/loop-workflows`) + shell (`host-workflows`) | Dual implement/validate workers |
| GitHub as only **proven** instance | Sandcastle / sandbox orchestrator adapter |
| Three packages: `init-workflows`, `loop-workflows`, `host-workflows` | Matt (or other) planning skill bundling |
| Progress `outcome:` control plane (no `last-tick.json`) | Stack/epic/merge bots; unbounded drain without explicit N |
| One-line spawn files (not full `.agent-workflows/config`) | Machine-readable policy config |

## Layers / mental model

```text
v0.1  /init-workflows     → contracts (docs/agents + .agent-workflows)
v0.2  /loop-workflows     → GitHub-only once | max N in one session
v0.3  policy ops + tick   → shared tick; chat workers for multi-N; thin shell host
later host runner fleet   → park, multi-spawn, rate-limit, stack engines
```

| Layer | Role |
|-------|------|
| **0 — Contracts** | `init-workflows` ensures policy + runtime; seeds encode ops contract |
| **1 — Planning** | Optional upstream (Matt / manual). Not bundled |
| **2 — Autonomy** | Worker tick in `loop-workflows`; schedulers only count N and stop |

## Packages

| Unit | Role |
|------|------|
| **`init-workflows`** | Contracts first. **Offer** (confirm) `host-workflows` install + spawn interview. Does **not** force-install `loop-workflows`. |
| **`loop-workflows`** | Sole home of the shared tick recipe **and** chat entry. Rewrite in place (keep name). |
| **`host-workflows`** | Thin sequential shell: `SKILL.md` + `scripts/host.sh`. Install never executes scripts. |

Hub layout: all three under `skills/`; discoverable by `npx skills add`.

### Breaking change (0.2 → 0.3)

- **`/loop-workflows max N`**: parent **only schedules**; each tick is a **fresh one-tick worker** (subagent). No more N implements stuffed into one session context.
- **once** (default): still one tick **in this session**.
- No compatibility flag — pin 0.2 install if old multi-N is required.

## Shared tick

Worker-owned, both entry paths:

```text
resume | pick → claim → implement → publish → progress
```

Skill soft-skip walk (after thin **list-queue**):

- **detect-publish-artifact** present → SOFT_SKIP
- open blockers (via **read-ticket**) → SOFT_SKIP
- skill-side: Matt-style **spec/PRD** bodies → re-label ready-for-human, outcome SKIPPED, no artifact
- queue non-empty but nothing claimable → BLOCKED

Implement quality (unchanged intent from 0.2): typecheck + focused tests + broader pass before publish when the product has them; prefer repo quality gates (not free-form shell from issue/comment text); account for AC.

## Dual schedulers

| Entry | Role |
|-------|------|
| **Chat** `/loop-workflows` | once = one in-session tick; max N = parent schedules N fresh one-tick workers |
| **Shell** `host-workflows` / `scripts/host.sh` | Sequential 1..N: spawn one-shot agent → read progress → stop rules |

Both run the **same** tick recipe. Paths differ only in who schedules.

## Host–worker handshake

| | Worker (agent session) | Scheduler (chat parent or shell) |
|--|------------------------|----------------------------------|
| Owns | Full tick including pick/resume | Outer N and stop rules only |
| Input | Product cwd + “exactly one shared tick” + discover `docs/agents/*` | — |
| **Not** passed | issue id, remaining N, host queue blob, stack base | — |
| Control plane | Append progress with `outcome:` | Re-observe queue; grep latest `outcome:` |
| Process exit | Not task success | Do not drive loop on exit alone |

No `last-tick.json`. Workers do **not** preload full `progress.md` by default (append only).

### Scheduler stop rules

| Condition | Action |
|-----------|--------|
| Empty ready-queue + no incomplete claim | COMPLETE (prefer detect **before** spawn) |
| Latest progress `outcome:` is BLOCKED, HARD_STOP, or FAILED | Stop |
| Tick SHIPPED / NEEDS_INFO / SKIPPED and work remains | Continue if `i < N` |
| Hit N with work left | MAX |
| Missing/unusable progress after a spawn that should have written it | FAILED / stop |

## Spawn (shell)

Resolve command string:

```text
--spawn / AGENT_SPAWN  >  product .agent-workflows/spawn  >  machine ~/.config/agent-workflows/spawn
```

- All missing → clear error / HARD STOP (no silent wrong binary).
- Run: `$SPAWN "<tick prompt>"` — prompt is the **final** CLI argument.
- Host never adds `--continue` / `--resume`.
- Unattended flags live in the **human-owned** spawn string only (not skill text).
- One-line spawn files only — **not** full `.agent-workflows/config`.
- No universal hub default binary; README may list example recipes.

## Tracker ops contract

Product file: `docs/agents/issue-tracker.md`.

Skills call **op names** and triage **role names** only. Concrete CLIs live in policy Steps. Role strings map via `docs/agents/triage-labels.md`.

### Required ops (each: Input / Steps / Success / Failure)

1. **preflight**
2. **integration-base**
3. **list-queue** — thin ready list, oldest first (not pre-filtered claimable)
4. **read-ticket** — incl. how to read open blockers
5. **comment**
6. **label-transition**
7. **claim**
8. **detect-publish-artifact**
9. **incomplete-claim**
10. **create-publish-artifact**

### Failure vocabulary (skill + policy + host)

| Word | Meaning |
|------|---------|
| **HARD_STOP** | Can't use tracker/env safely — scheduler stops |
| **SOFT_SKIP** | This ticket not takeable — try next; none left → BLOCKED |
| **NEEDS_INFO** | Can't finish cleanly — label + progress; no success publish |
| **OK** | Op succeeded |

### Seeds

| Seed | Bar |
|------|-----|
| GitHub (`issue-tracker-github.md`) | All ten ops **filled** — reference instance |
| Local (`issue-tracker-local.md`) | Same headings; stub bodies OK in 0.3 |

Structural check: `scripts/check-ops-contract.sh` (fails if a required GitHub op/field or local heading is missing).

### Claim / publish product meaning

- **Claim** = leave-queue only: claim comment + remove ready-for-agent. No `claimed` role. Race → SOFT_SKIP; infra → HARD_STOP. Mid-flight inferred via incomplete-claim.
- **Publish artifact** = durable review handoff (GitHub: non-draft PR to integration base, linked to ticket). Agent does **not** merge or close-as-done.
- Success: create-publish-artifact → label-transition to ready-for-human → outcome **SHIPPED**.
- Fail: comment → needs-info → no artifact → outcome **NEEDS_INFO**. Never re-apply ready-for-agent (human re-queues).
- **ready-for-human** = agent done for now; artifact optional (absent on spec skip).
- **label-transition**: exactly one of the five canonical roles; non-triage labels untouched.
- Spec/PRD detection remains **skill-side** after read-ticket.

## Progress control plane

Path: `.agent-workflows/progress.md` (auto-create from template if missing; never wipe existing body).

Minimal machine-readable entry:

```markdown
### YYYY-MM-DD — #N — <title>
- **outcome:** SHIPPED | NEEDS_INFO | SKIPPED | COMPLETE | BLOCKED | HARD_STOP | FAILED
- **publish:** <url or none>
- **checks:** pass | fail | n/a
- **note:** ≤1 line
```

| `outcome:` | When |
|------------|------|
| **SHIPPED** | Publish artifact created; ready-for-human |
| **NEEDS_INFO** | Fail path; needs-info label; no success artifact |
| **SKIPPED** | Soft-skip that settled the ticket (e.g. spec/PRD → ready-for-human) |
| **COMPLETE** | Empty queue + no incomplete claim (scheduler terminal) |
| **BLOCKED** | Queue non-empty; nothing claimable |
| **HARD_STOP** | Preflight / env / infra failure |
| **FAILED** | Missing progress after spawn, or unusable control-plane signal |

Optional prose fields may remain if compatible; **`outcome:` is required** for host/scheduler control.

## Init (0.3 delta)

- Seeds carry full ops contract (foundation).
- **Offer** host-workflows install + spawn interview (confirm; not force; product / machine / flag-only).
- Does **not** force-install `loop-workflows`.
- Re-run repairs/augments without wiping progress.

## Dogfood / definition of done (0.3 ship)

| Tier | Criteria |
|------|----------|
| Hub foundation | This freeze + ops seeds + progress contract + structural check |
| Hub ship | + loop rewrite + host-workflows + init offer + README/CHANGELOG |
| Once | Clean product tree: claimable ready issue → checks → artifact + ready-for-human + progress |
| Multi-N | Explicit N≥2 via workers (chat or shell), not one stuffed context |
| Negatives | Dirty hard stop; missing spawn HARD STOP; empty COMPLETE; check-fail → needs-info |

## Explicit out of scope

- Sandcastle or any sandbox orchestrator adapter
- Bundling Matt (or other) planning skills into this hub
- Stack/epic base engines and merge bots
- Replacing or archiving other skill repos
- Unbounded drain without explicit N owned by the host/scheduler
- Full local-markdown or Linear **runtime** proof in 0.3
- Park-branch / dirty-tree recovery fleets beyond thin shell stop rules
- Dual implement/validate workers
- Multi-spawn fleet and rate-limit backoff
- Machine-readable `.agent-workflows/config` beyond one-line spawn files
- Changing `/to-spec` to stop applying `ready-for-agent` (loop soft-skips PRDs instead)
